# База знаний дошкольника

Мобильное приложение предназначено для подбора развивающих заданий для детей дошкольного возраста, просмотра карточек упражнений, ведения профилей детей, сохранения истории занятий и обновления содержимого через серверную часть. Проект реализует подход, при котором основная работа приложения выполняется автономно на устройстве пользователя, а сервер используется для централизованного обновления контента.

## Технологический стек

- **Dart и Flutter** — разработка мобильного клиента и интерфейса приложения.
- **Android Studio** — основная среда разработки, запуска, тестирования и отладки мобильной части.
- **SQLite** — локальная база данных на устройстве пользователя.
- **SQL** — описание структуры таблиц, связей и запросов к данным.
- **Python** — серверная часть для выдачи обновлений контента.
- **JSON** — формат обмена данными между сервером и мобильным приложением.
- **HTTP API** — механизм получения актуального набора заданий.
- **HTML5 и CSS3** — использовались на этапе прототипирования экранов приложения.
- **Git** — система контроля версий для сопровождения разработки.

В мобильном приложении используются Flutter-пакеты:

- `sqflite` — работа с SQLite;
- `path` и `path_provider` — формирование путей к локальной базе;
- `http` — выполнение HTTP-запросов к серверу обновлений.

Серверная часть написана на Python с использованием стандартной библиотеки, поэтому для запуска сервера не требуется установка дополнительных Python-пакетов.

## Реализация мобильной части

Мобильный клиент находится в папке `mobile_app`. Приложение создано на Flutter и реализовано как Android-приложение с Material-интерфейсом. Основной файл интерфейса — `lib/main.dart`.

В приложении используется единая оболочка экранов `PageShell`, которая задаёт общий внешний вид: верхнюю строку состояния, заголовок, подзаголовок, мягкий пастельный фон, карточки, кнопки, чипы и нижнюю навигацию. Это позволяет сохранить единый стиль, соответствующий HTML-макетам из папки проекта.

Основная навигация реализована через `MainShell` и нижнюю панель `NavigationBar`. Пользователь может переходить между тремя ключевыми разделами:

- **Главная** — рекомендации, выбранный ребёнок и быстрые действия;
- **Каталог** — список упражнений, поиск, фильтры и сортировка;
- **Профиль** — данные детей, цели развития и история занятий.

Дополнительные экраны открываются через стандартные переходы Flutter:

- `ActivityDetailScreen` — карточка выбранного упражнения;
- `DoneScreen` — сохранение результата выполнения;
- `SyncScreen` — получение обновлений с сервера.

Интерфейс построен с использованием стандартных Flutter-виджетов: `MaterialApp`, `Scaffold`, `NavigationBar`, `TextField`, `DropdownButton`, `Slider`, `FutureBuilder`, `AlertDialog`, `InkWell`, `Container`, `Column`, `Row`, `Wrap`. Данные загружаются асинхронно из SQLite, поэтому экраны используют `FutureBuilder` и обновляют содержимое после пользовательских действий.

## Реализация локальной базы данных

Работа с SQLite сосредоточена в файле `mobile_app/lib/app_database.dart`. При первом запуске приложение создаёт локальную базу `preschool_knowledge.db` во внутреннем каталоге приложения.

В базе используются следующие таблицы:

- `activities` — развивающие упражнения;
- `age_groups` — возрастные группы;
- `activity_age_groups` — связь упражнений с возрастными группами;
- `domains` — направления развития;
- `activity_domains` — связь упражнений с направлениями;
- `tags` — теги упражнений;
- `activity_tags` — связь упражнений с тегами;
- `children` — профили детей;
- `activity_logs` — история выполненных или пропущенных занятий;
- `media` — сведения о медиафайлах;
- `activity_media` — связь медиа с упражнениями;
- `sync_state` — текущее состояние синхронизации;
- `sync_sessions` — история попыток обновления.

Структура базы соответствует реляционной модели. Связи между сущностями реализованы через внешние ключи, например упражнение может относиться к нескольким направлениям развития и нескольким возрастным группам. Для целостности данных используются ограничения `CHECK`, первичные ключи и уникальные поля.

При первом запуске база автоматически наполняется стартовыми данными:

- направления развития: речь, внимание, мелкая моторика, логика, память, сенсорика;
- возрастные группы от 3 до 7 лет;
- демонстрационный профиль ребёнка;
- набор встроенных упражнений.

В таблице `activities` предусмотрены поля `remote_id`, `version`, `source`, `updated_at` и `is_deleted`. Они нужны для синхронизации локальной базы с сервером. Поле `source` показывает происхождение записи: встроенное задание или удалённое обновление. Поле `version` позволяет обновлять только актуальные версии записей, а `is_deleted` используется для логического удаления контента.

## Поиск, фильтрация и история занятий

Каталог упражнений получает данные из SQLite через метод `getActivities`. Запрос строится с учётом выбранных параметров:

- текстовый поиск;
- выбранное направление развития;
- возраст ребёнка;
- режим коротких занятий;
- выбранный способ сортировки.

Поиск выполняется не только по названию упражнения, но и по описанию, инструкции, материалам, тегам и направлениям развития. Это делает каталог удобным для пользователя: можно искать как по теме, так и по конкретному навыку.

После выполнения упражнения пользователь открывает экран отметки результата. Приложение сохраняет запись в таблицу `activity_logs`, где фиксируются:

- профиль ребёнка;
- выбранное упражнение;
- дата и время;
- статус занятия;
- оценка;
- комментарий.

История занятий отображается в профиле ребёнка и на главном экране. За счёт этого пользователь может видеть, какие упражнения уже выполнялись, и отслеживать динамику работы.

## Реализация синхронизации

Синхронизация реализована в файле `mobile_app/lib/sync_service.dart`. Пользователь открывает экран обновлений и запускает проверку. Приложение отправляет HTTP-запрос на сервер:

```text
GET /api/content
```

По умолчанию для Android-эмулятора используется адрес:

```text
http://10.0.2.2:8000
```

Это специальный адрес, через который Android-эмулятор обращается к локальному компьютеру. При запуске на физическом устройстве вместо него необходимо указать IP-адрес компьютера в локальной сети.

Сервер возвращает JSON с актуальными направлениями, возрастными группами и упражнениями. После получения ответа приложение:

1. декодирует JSON;
2. проверяет наличие `remote_id` у удалённых упражнений;
3. сравнивает локальную и серверную версии записи;
4. добавляет новые упражнения;
5. обновляет существующие упражнения при увеличении версии;
6. помечает удалённые записи через `is_deleted`;
7. пересоздаёт связи упражнения с тегами, возрастами и направлениями;
8. обновляет таблицу `sync_state`;
9. сохраняет запись о сеансе синхронизации в `sync_sessions`.

Такой механизм позволяет обновлять содержимое базы знаний без переустановки приложения и без потери пользовательских данных, таких как профили детей и история занятий.

## Реализация серверной части

Серверная часть находится в папке `server`. Основной файл — `server.py`.

Сервер написан на Python и использует стандартные модули:

- `http.server` — создание HTTP-сервера;
- `ThreadingHTTPServer` — обработка запросов;
- `json` — чтение и выдача JSON-данных;
- `argparse` — настройка хоста, порта и пути к файлу контента;
- `pathlib` — работа с путями файловой системы.

Контент хранится в файле `server/content.json`. В нём описаны:

- версия набора данных;
- список направлений развития;
- список возрастных групп;
- список упражнений;
- теги;
- материалы;
- инструкции;
- сложность;
- длительность;
- связи упражнений с возрастными группами и направлениями.

Сервер предоставляет эндпоинты:

```text
GET /api/health
```

Возвращает состояние сервера и используется для проверки доступности.

```text
GET /api/content
```

Возвращает полный набор данных для обновления мобильного приложения.

```text
GET /api/updates
```

Алиас для получения обновлений. Добавлен для удобства расширения API.

Сервер также отдаёт CORS-заголовки, что упрощает тестирование API из разных клиентов. В текущей реализации сервер используется как централизованный источник контента, а не как постоянная удалённая база данных. Это соответствует архитектуре проекта: мобильное приложение работает с локальной SQLite-базой, а сервер нужен только для периодического обновления материалов.

## Подготовка Python-окружения

Команды выполняются из корневой папки проекта. Виртуальное окружение создаётся в каталоге `.venv`, поэтому интерпретатор и устанавливаемые Python-пакеты будут находиться рядом с проектом и не будут смешиваться с другими Python-проектами.

Каталог `.venv` не включается в репозиторий и не переносится между компьютерами. На новом компьютере его необходимо создать заново. Для создания окружения требуется один раз установить Python, после чего сервер запускается через локальный файл `.venv\Scripts\python.exe`.

Создание окружения в Windows:

```powershell
py -3 -m venv .venv
```

Если команда `py` отсутствует, но Python добавлен в `PATH`:

```powershell
python -m venv .venv
```

Активация окружения в PowerShell:

```powershell
.\.venv\Scripts\Activate.ps1
```

Установка зависимостей сервера:

```powershell
python -m pip install --upgrade pip
python -m pip install -r .\server\requirements.txt
```

В текущей версии сервер использует только стандартную библиотеку Python, поэтому список внешних зависимостей пуст. Файл `requirements.txt` сохранён для единообразного развёртывания и последующего расширения сервера.

Если PowerShell запрещает запуск скрипта активации, для текущего пользователя можно разрешить выполнение локальных скриптов:

```powershell
Set-ExecutionPolicy -Scope CurrentUser RemoteSigned
```

Вместо активации окружения сервер можно запустить напрямую через локальный интерпретатор:

```powershell
.\.venv\Scripts\python.exe .\server\server.py --host 0.0.0.0 --port 8000
```

## Запуск сервера

После активации `.venv` из корневой папки проекта:

```powershell
python .\server\server.py --host 0.0.0.0 --port 8000
```

Сервер должен оставаться запущенным во время проверки обновления контента в мобильном приложении.

Проверка сервера:

```powershell
Invoke-RestMethod "http://127.0.0.1:8000/api/health"
Invoke-RestMethod "http://127.0.0.1:8000/api/content"
```

## Запуск мобильного приложения

1. Запустите Android-эмулятор в Android Studio или подключите физическое устройство.
2. Запустите Python-сервер обновлений в отдельном терминале.
3. Из корневой папки проекта выполните:

```powershell
cd .\mobile_app
flutter pub get
flutter run
```

Если подключено несколько устройств, Flutter предложит выбрать нужное. Также устройство можно указать явно:

```powershell
flutter devices
flutter run -d <device-id>
```

Для Android-эмулятора адрес сервера в приложении:

```text
http://10.0.2.2:8000
```

Адрес `10.0.2.2` используется Android-эмулятором для обращения к компьютеру, на котором запущен сервер.

При запуске на физическом устройстве необходимо:

- подключить телефон и компьютер к одной локальной сети;
- узнать локальный IP-адрес компьютера;
- указать в приложении адрес вида `http://192.168.1.100:8000`;
- разрешить входящие подключения к порту `8000` в брандмауэре, если они блокируются.

Для сборки APK:

```powershell
cd .\mobile_app
flutter build apk --debug
```

Собранный debug APK находится по пути:

```text
mobile_app/build/app/outputs/flutter-apk/app-debug.apk
```
